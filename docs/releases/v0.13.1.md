# Release v0.13.1

Extends the `@env(...)` macro so it can be embedded inside bare `.ntd` values —
resolving and splicing into the surrounding text instead of being stored
literally — alongside documentation updates and a dependency audit.

- Extended `@env(...)` so it can be embedded inside a bare (unquoted) `.ntd`
  value instead of only standing alone: `path: /todos/@env(id or 1)` now resolves
  and splices into the surrounding text, compiling to `/todos/1` when `id` is
  unset and `/todos/<value>` when it is set. The definition grammar tokenizes a
  bare value into literal text and embedded macros, so the macro's `or` default
  is parsed structurally rather than left as literal characters.
- Kept the type distinction between the two positions — a standalone `@env(...)`
  value keeps its native type (a numeric default stays a number), while an
  embedded macro is stringified into the surrounding value.
- Limited embedding to bare values: inside a quoted string the `@env(...)` text
  stays literal (`"/todos/@env(id)"` is stored verbatim), and a non-macro `@`
  such as `/users/@me` is preserved unchanged.
- Documented embedded `@env(...)` across the README and the language-runtime and
  project-editor skills, including the bare-only rule and the quoted-string and
  literal-`@` caveats.
- Added compiler test coverage for embedded macros — default fallback,
  `process.env` resolution mid-path, string defaults, and literal `@` — with a
  new `embedded-env.ntd` fixture.
- Refreshed the dependency lockfile through a package audit, pulling patch
  updates to the Babel toolchain and other transitive packages (including
  `form-data`).
