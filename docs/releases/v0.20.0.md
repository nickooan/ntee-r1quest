# Release v0.20.0

Introduces joint chain files: a new kind of `.nts` file that chains other
requests into one run. Declare `@joint()`, list `-> @pick(...) -> @run(...)`
steps, and execute the whole chain as a single one-shot command or straight
from the terminal app — each step feeds picked response values into the next,
every call lands in history under one trace id, and only the final response is
printed. The editor keeps up: syntax highlighting and completions understand
the new grammar, and file rendering no longer rescans the whole buffer on
every frame.

- **Joint chain files (`@joint` / `->` / `@pick` / `@run`).** A `.nts` file
  that declares `@joint()` after its `ref` lines becomes a chain instead of a
  request — the grammar makes the two mutually exclusive, so
  `url`/`type`/`header`/`auth`/`body` cannot appear in a joint file. Steps run
  strictly in order: `@pick(key: <source>, ...)` merges values into the chain
  env that later steps read via `@env(...)` in their ref'd `.ntd` files, and
  `@run(<path>)` executes another `.nts` (path relative to the joint file,
  extension optional). Pick sources are a json path into the previous step's
  response body (`data.user.posts.data[0].id` — dots and `[n]` only) or
  `@i(key)` (with `or` defaults) from the joint file's own refs; values
  accumulate across steps and later picks win on duplicate keys. The trace id
  may be single- or double-quoted, or omitted to auto-generate one; the CLI
  `-ti` flag overrides it and `-env` seeds the chain env. Rule of `@joint`:
  every step must be an `application/json` request with a JSON response, and a
  joint file cannot `@run` another joint file. A working example ships at
  `example/request/queries/query-user-post.joint.nts`.
- **One command, one trace, one response.** `r1q -r <root> -p <chain>` runs
  the whole chain: intermediate responses are not printed — the final response
  renders with the trace id plus a `Joint chain: N steps completed` footer
  pointing at `@h <trace-id>`, which lists every step in call order. A failing
  step stops the chain (`Joint step N/M (<target>) failed.`, exit 1) with the
  failed response rendered in full; the steps that already ran are in history.
  While the terminal app is open, a CLI-run chain still persists every step
  (the call record rides the external event, now tagged `intermediate` so the
  app's results pane doesn't flip mid-chain — only the final or failing step
  shows).
- **Chains run from the terminal app too.** Select or type a joint file in
  `@query` mode and press Enter like any request: the chain executes in the
  runtime process (every step recorded directly), and the results pane shows
  the final response with the trace id and step-count footer — or the failing
  step's response under a `Joint step N/M failed.` banner. `ExecuteResult`
  gains optional `traceId` / `stepCount` / `failedStep` fields (older TUI
  binaries simply ignore them), and a pending guard now drops a second Enter
  while a run is in flight — the runtime mutates global cwd/env state per run,
  and longer chains made that race easier to hit.
- **Highlighting learns the chain grammar — and gets faster.** `->` arrows
  render cyan bold like keywords; `@joint`/`@pick`/`@run` get the familiar
  red-`@`/green-action macro colors; pick keys are cyan, json paths blue,
  `@run`/`ref` targets yellow, single-quoted trace ids yellow. Nested macro
  args (`@pick(content: @i(content-type))`) are matched by a balanced-paren
  scan, since a regex can't span them. The palette also gets a general polish:
  object keys (`userId:`) in `.ntd` files and bodies are cyan, the method
  after `type` is magenta bold, and `{}[],:` punctuation dims to gray so JSON
  bodies read with structure receding behind values. On performance, the
  whole-file GraphQL-block scan that ran on every frame (every cursor move and
  scroll) is now computed once per buffer change — tracked by an editor
  revision counter — and per-segment styles are memoized; only the visible
  rows are highlighted per frame.
- **Completions adapt to what you're editing.** In a joint buffer the pool
  swaps to chain statements: `-` on a new line offers the `-> @run()` /
  `-> @pick()` step templates (cursor landing inside the parens), `@` offers
  `@joint`/`@pick`/`@run`/`@i` plus `@i(key)` values from ref'd `.ntd`s, and
  `@run(` completes sibling `.nts` scripts extension-less, excluding the file
  being edited — request keywords disappear since they can't parse there. New
  request-file completions: `type ` lists HTTP methods, `auth ` lists
  `bearer`/`basic`, and `@f(` completes upload file paths. Two correctness
  fixes ride along: `@env` is no longer offered in `.nts` buffers (it's a
  `.ntd`-only macro — a compile error elsewhere) and `.ntd` buffers get their
  own pool (`@env` + custom entry keys) instead of request keywords. Each
  overlay row now shows a faint kind tag (`macro`, `header`, `definition`, …)
  when the pane is wide enough.
- **AI plugin skills know how to chain.** The one-shot-runner skill's
  chained-requests workflow now prefers generating a temporary joint file at
  the collection root and running it as one command — one process, all steps
  auto-traced — with the old per-step `-ti`/`-env` orchestration demoted to a
  fallback for flows that need per-step confirmation or non-JSON steps. The
  language-runtime skill documents the joint syntax and rules.
