# Release v0.16.1

Hardens the new storage engine for real multi-process use and takes the last
big cost off its write path: the history store now has a single-writer lock,
one-shot runs hand their call records to an open terminal app over the event
socket (so history stays complete either way), and periodic index snapshots
moved to a background goroutine — making cached writes hundreds of times
faster. Plus a TUI rendering fix.

- **Single-writer lock on the store.** Opening the database now takes an
  exclusive kernel lock (`flock`); a second process fails fast and degrades to
  a cache-less run instead of risking corruption. The lock is tied to the
  process, not a file on disk — it releases automatically on any exit
  (Ctrl+C, crash, `kill -9`), so no stale-lock state is possible.
- **One-shot history hand-off to an open app.** With `sock` configured, a `-p`
  run now sends its **full call record** over the event socket — the open
  terminal app (which holds the store lock) persists it and highlights the
  request, so history stays complete and live while the app is open. Failed
  (non-2xx) calls are handed off too, and a missing/dead socket stays quiet.
  `r1q --init` now defaults `sock` to a path under the OS temp directory.
- **Cache writes are ~360× faster.** The periodic index-hint rewrite (a full
  index serialization plus two fsyncs, previously paid synchronously every few
  writes) now runs in a background goroutine behind a consistent snapshot — a
  cached write dropped from ~1.3 ms to a few microseconds. Close, compaction,
  and range deletes remain synchronous checkpoints; verified race-clean.
- **Fixed: requests run from the terminal app were not saved to history.** The
  CLI entry held the store lock through the whole session after its startup
  housekeeping, locking out the app's own runtime; it now releases the lock
  before the app launches.
- **Fixed: `�` characters in search mode.** The search renderer truncated
  lines by bytes, splitting multi-byte characters (like the `─` section
  dividers); it now truncates by runes.
- **Measured performance docs.** The database binding's README now carries a
  reproducible benchmark against lmdb (reads, sync/async writes, indexed
  queries) and a "why r1quest uses ntee-db" summary — including where lmdb is
  faster and why that doesn't matter for a local, human-paced app.
