# Release v0.19.0

Turns the in-place editor into a real editor. Edit mode gains selection,
undo/redo, in-buffer search that moves the cursor, horizontal scrolling for long
lines, and header-value completions — with a per-file edit history persisted in
the embedded `ntee-db` cache so heavy content never sits on the TUI heap. Under
the hood, the `ntee-db` read surface finishes going async: every read now runs
off the event loop.

- **Edit mode grows real editing controls.** A long line now scrolls
  horizontally so the cursor stays visible instead of sticking at the right
  edge. `Ctrl+A` selects the word under the cursor and, on repeat, grows to the
  `key:` / value segment (the `:` stays with the key) and then the whole line;
  typing or Backspace over a selection replaces it; `Esc` clears the selection
  before it discards. The status line shows a yellow **`editing`** badge while
  there are unsaved changes and a green **`saved`** badge once the buffer matches
  disk.
- **Undo / redo, backed by a file-version history.** `Ctrl+Z` / `Ctrl+Y` walk a
  per-file timeline coalesced by edit burst (a new edit after an undo drops the
  redo branch). Snapshots live in the runtime's `ntee-db` cache — a new `file`
  secondary index capped at 50 versions per file, oldest auto-evicted — so the
  content stays off the TUI heap and the editor keeps only a lightweight seq
  list plus a cursor. The TUI reaches the store over new `snapshot/*` RPC
  methods (put/get/list/delete); saves are recorded as `save` checkpoints, edit
  bursts as `edit`.
- **In-buffer search that moves the cursor.** `Ctrl+F` searches the file you're
  editing; Enter commits by moving the editor cursor to the focused match —
  including a match scrolled off the right edge, which the search pane now
  scrolls to horizontally. Previously search was view-only and never touched the
  editor cursor.
- **Header-value completions.** After `header <name>,` the completion overlay now
  suggests common values for that header — `content-type`
  (`application/json; charset=utf-8`, `multipart/form-data`, …), `authorization`
  schemes (`Bearer `, `Basic `, `Digest `, `AWS4-HMAC-SHA256 `, …),
  `cache-control` (`no-store`, `max-age=0`, …), `accept`, `accept-encoding`, and
  more — alongside the existing header-name completions. Values that carry `/`,
  `;`, or `,` complete as a whole, which the word-based name matcher could not
  do.
- **`ntee-db` reads are now fully async.** Completing the migration started in
  v0.18.0, the remaining point-scale methods (`get`, `has`, `stats`,
  `prefixScan`, `secIndex`, `secIndexHas`, `secIndexPrefix`, `secIndexRange`,
  `secIndexDropped`, `secIndexProspective`) are now `Promise`-returning as well.
  The read surface is uniform and non-blocking end to end, and independent reads
  run in parallel on worker threads (`Promise.all`); the trade is a worker-thread
  hop on each point read, negligible at the app's read volume. `put` / `delete`
  stay synchronous. The runtime cache layer awaits these throughout. The
  embedded binding moves to `@ntee/ntee-db` 0.4.0 — the one breaking change:
  direct callers must now `await` reads.
