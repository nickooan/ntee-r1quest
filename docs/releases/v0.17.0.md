# Release v0.17.0

Reworks the embedded history store (`ntee-db`) into a JSON-native cache and
takes the last redundant hop off its record reads. History behaves exactly the
same for users — the cache is just simpler and faster under the hood. The
embedded binding moves to `@ntee/ntee-db` 0.2.0.

- **The history store is now JSON-native.** Reads return records already parsed
  — no `JSON.parse` at the call site — and writes accept a plain object,
  serialized for you (no `JSON.stringify`). A binary or non-JSON value still
  round-trips, coming back as a `Buffer`. The app's cache layer
  (`cacheGet` / `cachePut`) sheds its manual encode/decode as a result.
- **Record-returning index searches are batched.** A new native `getMany`
  collapses "find the keys for an index value, then fetch each record" from N+1
  FFI crossings into a single call; `secIndexRecords` and the other `*Records`
  searches use it (the 20-match History-list fetch dropped from ~116 µs to
  ~85 µs).
- **Clearer secondary-index API, plus a presence check.** The index methods
  were regrouped under a `secIndex` prefix — `byIndex` → `secIndex`,
  `searchByIndex` → `secIndexRecords`, and so on — and a new `secIndexHas`
  answers "does any record carry this value?" without materializing the matching
  keys (the secondary-index counterpart of `has`).
- **Benchmarked against SQLite as well as lmdb.** The binding's README now
  measures ntee-db against `better-sqlite3` and `lmdb`, tier by tier (sync
  writes, point reads, indexed queries), with an honest read of where each wins
  — ntee-db leads on caller-synchronous writes, sorted-key scans, and the
  grouped newest-per-value query the History list depends on.
